# Security Audit — 2026-04-16

**Auditor:** automated external-threat mapping + manual surface review
**Scope:** bazaar recipe overlay, its validator, its CI pipeline
**Bazaar HEAD at audit:** pending commit of this session's changes
**Method:** surveyed known 2024–2026 CVEs and real incidents in package-manager ecosystems (AUR, PyPI, npm, Arch, AGNOS adjacencies), mapped each attack class to bazaar's attack surface, rated likelihood × blast radius, triaged fixes

## Threat model summary

Bazaar accepts contributor-submitted recipes via GitHub PRs. Each recipe declares where to download source tarballs and what shell to run to build them. Merged recipes are installed by `ark bazaar install <pkg>` which downloads the source, verifies `sha256`, and runs the build inside a sandbox (no network access by default). Therefore the threats an attacker cares about are:

1. Landing a **malicious recipe** that passes review (typosquat, homoglyph, look-alike metadata, backdoor smuggled into trusted-looking source)
2. **Supply-chain substitution** of the upstream source after review (URL hijack, tag retargeting)
3. **Sandbox escape** at install time (not really bazaar's attack surface — belongs to ark/takumi — but we can minimize what recipes need to do inside the sandbox)
4. **CI pipeline compromise** (workflow injection, action tag retargeting, secret exfiltration)
5. **Validator bugs** that let malformed/malicious recipes pass the gate

## Findings

Severity uses CVSS-ish bands: **Critical** = unauth remote code execution on review/install, **High** = successful malicious merge with moderate effort, **Medium** = defense-in-depth weakness enabling the above, **Low** = theoretical or mitigated elsewhere.

### F1 — CI workflows do not pin third-party actions to SHA  (High, **FIXED**)

**Surface:** both workflows reference actions by tag. `validate-recipes.yml` uses `actions/checkout@v4`; `release.yml` uses `actions/checkout@v4` *and* `softprops/action-gh-release@v2`. The release workflow runs with `permissions: contents: write` and the `GITHUB_TOKEN` is in-scope — a retargeted third-party tag during release is the highest-value target in the repo.

**Real incident:** [CVE-2025-30066 (`tj-actions/changed-files`)](https://github.com/advisories/ghsa-mrrh-fwg8-r2c3) — an attacker retroactively modified `tj-actions/changed-files` tags (including `v45` series) to point at a malicious commit that exfiltrated CI secrets from >23,000 repos via log printing. `actions/checkout` is first-party and has better hardening, but pinning to a SHA is the industry-standard defense-in-depth. `softprops/action-gh-release` is third-party and the exact same risk class as `tj-actions`.

**Fix:** pinned all three usages to their current v-tag commits:
- `actions/checkout@34e114876b0b11c390a56381ad16ebd13914f8d5 # v4`
- `softprops/action-gh-release@3bb12739c298aeb8a4eeaf626c5b8d85266b0e65 # v2`

Bumping the tag requires updating both the SHA and the comment together, keeping drift visible in code review.

### F2 — CI job has default (read/write) token permissions  (Medium, **FIXED**)

**Surface:** no top-level `permissions:` block in the workflow means `GITHUB_TOKEN` defaults to read/write on contents, issues, PRs, etc. A compromised step could push commits, close PRs, or modify issues.

**Real incident:** implicated in multiple 2024–2025 GitHub Actions supply-chain attacks where malicious dependencies used permissive `GITHUB_TOKEN` to pivot. Mitigation: [GitHub Security Lab — Preventing pwn requests](https://securitylab.github.com/resources/github-actions-preventing-pwn-requests/).

**Fix:** added `permissions: contents: read` at workflow level. Validator + tests + bench only need read access.

### F3 — Validator accepts non-ASCII characters in `[package].name`  (High, **FIXED**)

**Surface:** A recipe named `nеovim` (with Cyrillic `е`, U+0435) would pass the validator — filename stem would be `nеovim.cyml` which visually matches but resolves as a different package at install time. This is the classic homoglyph attack against package ecosystems.

**Real incidents:**
- Ongoing campaigns on [PyPI](https://blog.checkpoint.com/securing-the-cloud/pypi-inundated-by-malicious-typosquatting-campaign/) using homoglyphs (Cyrillic `а` vs Latin `a`, Greek `ε` vs Latin `e`)
- [287 typosquatted npm packages](https://medium.com/@instatunnel/typosquatting-in-package-managers-the-attack-that-preys-on-a-single-keystroke-39588b832390) found in a single 2025 sweep; [500+ PyPI packages](https://blog.checkpoint.com/securing-the-cloud/pypi-inundated-by-malicious-typosquatting-campaign/) in a separate wave

**Fix:** validator now rejects `[package].name` and `[package].pkgbase` containing any byte > 0x7E. Allowed characters are ASCII letters, digits, `-`, `_`, `.`, `+`. Error message is specific so legit contributors hitting this (unlikely) can understand what failed.

**Residual risk:** ASCII-only still allows ASCII typosquats (`neovim` vs `necvim`). See F6.

### F4 — Validator does not enforce HTTPS for source URLs  (High, **FIXED**)

**Surface:** `[source].url = "http://example.org/..."` or `"ftp://..."` or `"file:///..."` would pass. HTTP allows MITM tarball swap; `file://` allows the local build host to be probed; `git+ssh://` allows credentials theft.

**Real incident:** sha256 pinning defends against remote tampering *if* the digest is populated, but recipes in drafting often have empty sha256 (validator warns but doesn't error). During that window, a non-HTTPS URL is a direct substitution vector.

**Fix:** validator now errors on any URL not starting with `https://`. No known bazaar recipe needs a non-HTTPS source; zugot's convention is the same.

### F5 — Shell metacharacters in `version` are not flagged  (Medium, **FIXED**)

**Surface:** `version = "1.0.0; rm -rf /"` combined with a `[build]` step that interpolates `$version` would yield command injection. Most recipes currently hardcode the version in the shell strings (`make -j$(nproc)` not `make VERSION=$version`) so this isn't directly exploitable today — but that's a convention, not an enforced constraint, and new recipes could expand.

**Real incidents:** PKGBUILD-style scripts have been a [recurring CVE class](https://portswigger.net/web-security/os-command-injection). `CVE-2023-51385` (OpenSSH) and many others show how user-controlled strings interpolated into shell without quoting lead to RCE.

**Fix:** validator warns if `[package].version` contains any of `; & | $ \` ' " backslash newline carriage-return` or starts with `-`. A warning rather than error lets edge cases through (e.g., a version literally containing `+`) while surfacing the attack pattern for reviewer scrutiny.

### F6 — Typosquatting detection is manual  (Medium, **UNFIXED — review responsibility**)

**Surface:** A PR introducing `neovim2` or `necvim` would pass all validator checks. Only a reviewer comparing against the existing recipe list catches the attack.

**Why not fixed now:** Levenshtein-distance-1 detection adds complexity and a false-positive rate (`neovim` vs `neo-vim` are deliberately different packages in many ecosystems). Better approach: a pre-merge bot that comments on new recipe names with the closest existing matches — out of scope for the validator binary.

**Tracking:** added to `docs/validator.md` roadmap.

### F7 — Empty `sha256` is a warning, not an error  (Medium, **UNFIXED — policy**)

**Surface:** Current validator warns on `sha256 = ""` but exits 0. A merge without a real digest means the install-time integrity check is whatever comes from `ark` (which currently has no backfill). Result: the first download wins, no verification.

**Why not fixed now:** intentional drafting affordance. A `--strict` mode that errors on empty sha256 is the right call, toggled on in CI only for PRs targeting main (not for draft PR iteration).

**Tracking:** added to `docs/validator.md` roadmap. Reviewer checklist already calls out "sha256 must be populated before merge."

### F8 — `${{ github.base_ref }}` interpolated into bash in the GPG step  (Low, **FIXED**)

**Surface:** The GPG-signature step uses `${{ github.base_ref }}` interpolated directly into a bash string. `base_ref` is populated from the PR target branch name — controlled by maintainers, not PR authors — so direct injection isn't possible today. But GitHub's own [workflow injection guidance](https://github.blog/security/vulnerability-research/how-to-catch-github-actions-workflow-injections-before-attackers-do/) recommends never interpolating context into `run:` blocks regardless of source.

**Fix:** passed via `env:` and referenced as `$BASE_REF` inside the shell, which shell-escapes properly.

### F9 — Cyrius install script fetched over HTTPS with no checksum  (Low, **ACCEPTED**)

**Surface:** CI does `curl -sSfL … install.sh | sh`. This is trust-on-first-use; a Cyrius release-pipeline compromise could push a malicious installer.

**Why accepted:** the toolchain needs to come from *somewhere*, and pinning a hash per release is coupled to the cyrius release cadence. Bazaar pins the `CYRIUS_VERSION` variable (currently `5.2.0`) which the installer is expected to honor. Longer-term: once cyrius ships signed release artifacts (sigstore or PGP), pin to those.

**Tracking:** ADR-002 roadmap.

### F10 — Stdlib TOML parser size limit (256KB) could be a DoS vector  (Low, **ACCEPTED**)

**Surface:** `lib/toml.cyr` allocates a fixed 256KB stack buffer in `toml_parse_file`. Files larger than 256KB are silently truncated. A recipe with lots of padding comments could be used to fit a benign-looking payload into the validator's 256KB while a real parser downstream sees the rest.

**Why accepted:** current largest recipe is ~5KB. Validator can add a size check if this becomes a real concern. Cyrius stdlib fix is the right long-term answer.

### F11 — No rate limit on recipe additions per PR  (Low, **ACCEPTED — policy**)

**Surface:** a single PR could add 500 malicious recipes. Review fatigue = merge.

**Why accepted:** `min_reviews: 1` in `MAINTAINERS.yaml` combined with `restricted_domains` mitigation (security/networking/ai require extra review). A per-PR cap would be mechanical but isn't a common attack in practice — the incentive structure favors subtle single-recipe attacks, not floods.

### Not applicable (for the record)

- **XZ-style social-engineering takeover (CVE-2024-3094)** — applies to upstream projects bazaar *references*, not bazaar itself. The correct mitigation is the review process + requiring a real sha256; both are in place. This is a risk we share with every downstream distribution.
- **Zip Slip / tarball path traversal** — tarball extraction happens in `ark`/`takumi`, not bazaar. Belongs in that audit.
- **vm2-style sandbox escape (CVE-2026-22709)** — not applicable. We don't run untrusted JS.
- **setuptools RCE via setup.py (CVE-2024-6345)** — applies at install-time in the build sandbox. Bazaar doesn't parse or run setup.py; it's an ark/takumi concern.
- **pickle deserialization (CVE-2025-1716)** — not applicable.
